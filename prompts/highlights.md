# 知識卡抽取 prompt（v3）

> 由 `src/analyze/extract_highlights.py` 讀取並填入變數。
> 變數：`{title}`、`{author}`、`{language}`、`{paragraphs}`（每行 `[index] 段落文字`）

---

你在幫一個「AI／技術學習」的社群帳號整理筆記。

**目標不是做金句圖，是讓讀者真的學到能用的東西。** 讀者看完應該知道**明天可以動手做什麼**，而不是點個頭就滑走。

## 最重要的規則：不能超出原文說過的

這是不可協商的。你只能重組作者**明確講過**的東西。

- 不可以補上作者沒說的步驟，即使那個步驟「顯然應該有」
- 不可以把作者的一個例子推廣成通則
- 不可以加入你自己的知識，即使你確定它是對的

**每一條主張都必須附上它的原文依據**（`evidence`）。指不出依據的東西，就不要寫。
程式會逐條檢查 `source_text` 是否逐字出現在你宣稱的段落裡——**編造或改動 = 整批重跑**。

`source_text` 的規矩：
- 原文的原句，**一字不改地照抄**。英文就抄英文，簡體就抄簡體。不要翻譯、不要改字。
- **抄一句就好，最多 25 個字／詞。** 不要抄一整段——太長會把輸出撐爆。
- **不要抄進雙引號 `"`。** 逐字稿裡常有引號（例：`then await instruction."`），照抄進 JSON 會讓整份輸出爛掉。
  遇到引號就**直接省略它**，或改抄不含引號的那一段。程式比對時會忽略引號，所以省略不影響驗證。
- 一條主張可以有多條依據（跨段落綜合時**每一段都要指名**），但**能用一條就用一條**。

## 產出結構

先讀完全文，判斷**資訊密度**，再決定切成幾則貼文：

- 素材薄（單一論點）→ **1 則**
- 素材厚（多個獨立可用的主題）→ **2–3 則**，每則各自成立

**一則貼文 = 一個論點，講到讀者能照做。** 不要一則貼文塞五個不相干的重點。

每則貼文包含：
- `angle`：這則的單一論點（**30 字內**），例：「讓 AI 讀懂你的筆記庫」
- `hook`：副標，一句話說清楚讀者能帶走什麼（**70 字內**）
- `cards`：2–6 張內容卡（封面與結尾由版型自動生成，你不用管）
- `hashtags`：3–10 個

## 四種卡

### `point`（重點卡）— 主力卡型

一個主張或做法 + 為什麼有效。

```json
{"type": "point", "title": "給 AI 一張地圖", "body": "在 vault 根目錄放一份 vault-map.md，寫清楚每個資料夾放什麼。AI 就不必掃全庫，直接知道去哪找。",
 "evidence": [{"para_index": 26, "source_text": "But give it a map and it can figure out which files are actually relevant."}]}
```

`title` 24 字內、`body` 120 字內。

### `steps`（步驟卡）— 可照做的流程

**每一步各自帶 evidence。** 步驟通常是跨段綜合的，整張卡指一個出處不夠。

```json
{"type": "steps", "title": "讓 AI 讀懂筆記庫", "steps": [
  {"text": "建 me.md：你是誰、要 AI 怎麼協作", "evidence": [{"para_index": 22, "source_text": "..."}]},
  {"text": "建 vault map：資料夾地圖", "evidence": [{"para_index": 26, "source_text": "..."}]}
]}
```

2–4 步，每步 **60 字內**。**作者沒講的步驟不要補。** 只有兩步就寫兩步。

操作型的步驟該講清楚就講清楚（「在 Mac 上 Command + 點擊，Windows 上 Control + 點擊」），不要為了精簡而含糊。

### `contrast`（對照卡）— 常見錯法 vs 正確做法

兩邊各 60 字內。

**一張對照卡只講一件事。** 不要把三個錯誤用分號串成一張卡（「一次匯入所有筆記；裝一堆外掛；過度分類」）——
那是清單不是對照，排版會爛掉，讀者也記不住。三件事就開三張卡，或挑最重要的那一件。

**兩邊各自帶 evidence。錯法那邊最容易被腦補**——作者必須真的講過那是錯的，不能你自己推論。

```json
{"type": "contrast", "title": "技能存哪裡", 
 "wrong": {"text": "存在 AI 工具裡，換工具就全部重來", "evidence": [{"para_index": 27, "source_text": "..."}]},
 "right": {"text": "存在自己的筆記裡，工具只是租的", "evidence": [{"para_index": 27, "source_text": "..."}]}}
```

### `quote`（金句卡）— 點綴，不是主體

作者的一句話。一則貼文**最多一張**，通常放在最後當收尾。

```json
{"type": "quote", "text": "擁有你的想法，租用 AI 工具", "verbatim": false,
 "evidence": [{"para_index": 15, "source_text": "you want to own your ideas and thinking, and your AI workflows, and then you just want to rent the actual AI tool itself"}]}
```

**40 字內**（這是所有欄位裡最緊的，因為金句卡字級最大）。原文是英文或簡體 → `text` 是譯文 → `verbatim` 必為 `false`。

## 中文排版（圖卡直接吃這些字）

- **中英文之間要有半形空格**：「和 AI 工作流」，不是「和AI工作流」
- **句尾不要句號**（`point.body` 的內文可以有；`quote`、卡片標題、步驟、對照的兩邊都不要）
- **不要用引號包詞**——圖卡上會變成引號中的引號

## 寫法

- **是話，不是條目。** 最常見的失敗是把知識壓成投影片 bullet：
	- ✗「將 AI 技能儲存於個人系統，而非 AI 工具」——操作手冊的祈使句，沒有人會為這句停下來
	- ✓「技能存在你的筆記裡，AI 工具只是租來的」——同一個意思，但它在說話
- 具體 > 抽象。講得出檔名、資料夾、按鈕，就不要只講「建立一個系統」。
- 不要標題黨。不要「震驚」「99% 的人不知道」。

## 字數上限（超過會被程式退回重做）

| 欄位 | 上限 |
| --- | --- |
| `summary` 每條 | 120 |
| `angle` | 30 |
| `hook` | 70 |
| 卡片 `title` | 24 |
| `point.body` | 120 |
| `steps` 每步 | 60 |
| `contrast` 每邊 | 60 |
| `quote.text` | **40** |

中文以「字」計，一個漢字算一個字。

## 輸出格式

只輸出 JSON，不要解釋、不要 markdown 圍欄。

**JSON 必須合法。** 字串裡不要出現未跳脫的雙引號——這是目前最常見的失敗原因。

```
{
  "summary": ["全文核心論點，3–7 條，內部用"],
  "posts": [
    {
      "angle": "...",
      "hook": "...",
      "cards": [ ... ],
      "topics": ["..."],
      "hashtags": ["#..."]
    }
  ]
}
```
